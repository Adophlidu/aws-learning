# Lambda 学习笔记

> 第 4 课的深入问答整理 ｜ 2026-06-27
> 围绕"它到底怎么工作"一路追问下来的理解，按"概念 + 关键纠错"组织。

---

## 一句话定位

**Lambda = 无服务器计算。你只上传一个函数，AWS 在事件触发时临时拉起来跑一次，跑完回收。平时不运行、不花钱。**

和 EC2 的本质区别：
- EC2：一个进程 7×24 常驻等请求 → 一直开机、一直花钱
- Lambda：平时不存在，请求来了才临时出现 → 按调用付费，不调用 0 元

---

## 1. 运行模型：单次执行 vs 服务寿命

- **15 分钟限制 = 单次"执行（invocation）"的最长时间，不是 API 的存活时间。**
- 你的 API 可以永久存在、被调用无数次；只是**任何一次**执行不能超过 15 分钟。
- 正常 API 请求几毫秒~几秒就返回，根本碰不到这个上限。
- 只有"单个请求要干很久的活"（大视频转码、百万行批处理）才会受限 → 那种该用 EC2 / Fargate / Batch / Step Functions。

### 为什么有 15 分钟限制（4 重原因）
1. **资源池高效流转**（最根本）：Lambda 是多租户共享资源池，靠"快进快出、高速复用"才便宜。长任务占着资源不放会拖垮调度。
2. **划清工具边界**：超过 15 分钟的活本来就不该用 Lambda，限制是一种设计引导。
3. **成本安全阀**：防止死循环 `while(true)` 永远跑、无限计费，最多 15 分钟强制刹车。
4. **运维可控**：任务有界，机器下线/回收时知道"最多等 15 分钟必结束"。

## 2. 触发机制：Lambda 不读我的代码

**核心纠错：Lambda 从不解析/扫描你的代码，它完全不懂你写了什么。**

- Lambda 是**事件驱动**的，靠**你显式配置的触发器（Trigger）**决定何时运行。
- 触发器例子：HTTP 请求来了 / S3 上传文件 / 定时 / 消息队列来消息。
- 没有触发器，函数永远不会运行。

### HTTP 路由（/test）不是 Lambda 管的，是 API Gateway 管的
```
用户访问 /test
   │
   ▼
API Gateway   ← 懂 HTTP、懂路由；你在这里配"/test → 哪个 Lambda"
   │ 把请求打包成 JSON 事件
   ▼
Lambda 函数   ← 只管"被调用就跑一次"，不管路由
```
- 路由是你在 API Gateway **手动配置**的，不是从代码里扫出来的。
- 用 Flask/Express 时两种做法：① API Gateway 每条路由配一个 Lambda；② 整个框架塞进一个 Lambda，用适配器（Mangum/serverless-http）在代码内部分发路由。

## 3. 语言无关的秘密：Handler 契约 + Runtime 分层

**Lambda 和我的代码之间只有一个极简契约：**
> 把一个 JSON 事件喂给你声明的入口函数 `handler(event, context)`，收下它的返回值。

- Lambda 不需要懂业务、不需要懂路由，只会"调用指定函数、喂 JSON、收返回值"。
- **语言不是检测出来的，是我在配置里声明的**（Runtime: Python 3.13 / Node.js 20 ...）。
- 既不是"上传时检测"，也不是"执行时检测"——是创建函数时的**配置元数据**。

### 执行环境是分层的
```
┌─ 执行环境（Firecracker 微虚拟机，做隔离） ─┐
│  你的代码 (app.handler)      ← 我写的       │
│  Runtime (如 Python 解释器)  ← 喂事件、调入口│
│  OS 基础环境 (Amazon Linux)  ← AWS 提供     │
└──────────────────────────────────────────┘
```
- Runtime 这层的职责：实现统一的 **Lambda Runtime API**——拉事件、调 handler、回传结果。
- 每种语言有自己的 Runtime 实现，但对外接口统一 → 所以天然支持多语言。
- 还支持**自定义运行时**：只要实现 Runtime API，任意语言（Rust/PHP/自创语言）都能跑。
- **Lambda 不内置所有语言，它只定义"运行时该怎么和我对接"的标准接口。**（关注点分离）

## 4. 内部架构：谁负责调度机器？

Lambda 内部分两层（云服务通用的控制面/数据面）：

```
① 控制面 Control Plane：管函数配置、存代码、管触发器映射
② 数据面 Data Plane：
   - 调用入口服务：接收"该跑了"
   - ★ 放置/调度服务：决定用哪台机器、要不要新建环境、并发大了多拉几个（扩缩容）
   - 工人机群 Worker Fleet：一堆物理机
```

**关键区分：**
| 谁 | 负责什么 |
|----|----------|
| Lambda 放置/调度服务 | 去哪台机器跑、跑几份、自动扩缩容 |
| Runtime 层 | 在已选好的机器内部，把事件喂给我的代码、跑一次 |

- "去哪台机器、跑几份" = **Lambda 调度**（数据面）。
- "在这台机器里怎么把事件交给代码" = **Runtime**。
- Runtime 对机器调度一无所知，它只活在被选中的那个盒子里。
- 我"不用管负载"的弹性，就是放置/调度服务实现的：1000 个并发请求 → 拉起 ~1000 个环境各跑一份；退潮后回收闲置环境。

## 5. 状态管理：Lambda 是无状态的

**持久 / 共享的状态必须放外部服务**：
- 数据库：DynamoDB、RDS
- 对象存储：S3
- 共享缓存：ElastiCache（Redis）
- 这些才是"状态的家"，Lambda 只是无状态计算。

### 重要细节：全局变量"有时"会活下来
- **热启动复用同一执行环境**时，函数外层（全局/模块级）变量会被保留。
- 但**绝不能依赖它**，因为：
  1. 冷启动 / 新环境时是初始状态
  2. 并发 N 个请求 = N 个独立环境，全局变量**互不相通**
  3. 环境随时被回收，全局状态随之消失
- 正确用法 = **尽力而为的性能缓存**，不是状态真相：
  - ✅ 复用 DB 连接、缓存配置 / SDK 客户端
  - ❌ 存购物车、累计计数等需要正确且一致的数据 → 必须进数据库
- `/tmp` 临时磁盘同理：热环境保留，只能当临时缓存。

> 区分：**全局变量 = 碰巧能复用的优化（不可靠、不共享）；DB/缓存服务 = 可靠、共享的状态真相。**

## 整体心智图

```
触发器(事件源) ──→ API Gateway(若是HTTP，负责路由)
                        │ JSON 事件
                        ▼
                 Lambda 控制面(配置) + 数据面(调度机器、扩缩容)
                        │ 选好机器、建好执行环境
                        ▼
                 执行环境: Runtime 把事件喂给 handler() → 跑一次 → 回收
                        │
                 状态去哪? ──→ 外部服务(DynamoDB/RDS/S3/Redis)
```

**贯穿全篇的设计思想：关注点分离 + 各司其职。** 学 EC2、容器时会反复见到同一套思想。
