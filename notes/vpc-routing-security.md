## 一句话核心

**子网没有内置的"公私"属性。公私由它关联的「路由表」决定——有没有一条 `0.0.0.0/0 → IGW`。** 整个 VPC 网络由四层拼成：**路由表(去哪) + 公网IP(能否用IGW) + NACL(子网级防火墙) + 安全组(资源级防火墙)。**

---

## 1. 路由表决定公私（不是创建时定死的）

- 创建子网时只填 CIDR + AZ，**没有公/私标签**。
- 每个子网关联一张**路由表**，路由表决定流量往哪走。
- 新建 VPC 的默认路由表只有一条：`10.0.0.0/16 → local`（VPC 内部互通）。

| 路由表配置 | 子网类型 | 与外网连通性 |
|------------|----------|--------------|
| `local` + `0.0.0.0/0 → IGW` | 公有 | 双向（可进可出）* |
| 只有 `local` | 私有（隔离） | 都不通，只在 VPC 内 |
| `local` + `0.0.0.0/0 → NAT` | 私有（可出网） | 单向（只出不进） |

- **动态**：随时改子网关联的路由表 → 公私即切换。不是创建时定死。
- 自建 VPC 子网**默认全私有**；账号自带的**默认 VPC** 已预配 IGW+路由，所以子网开箱即公有。
- "公有子网/私有子网"只是**约定俗成的称呼**，AWS 内部没有 type 字段。

\* 公有子网"双向"是有前提的，见第 3 点。

---

## 2. IGW 的真相：只为"有公网 IP 的资源"工作

- IGW（Internet Gateway）= VPC 通往公网的大门。
- 它干的是 **公网IP ↔ 私有IP 的一对一转换（NAT）**，**只对已分到公网 IP 的资源做**。
- 公网世界**不认识私有地址**（10.x / 172.16-31.x / 192.168.x 公网不可路由）。只有私有 IP 的资源，包发出去对方**没法回**。
- 所以："有 IGW 路由"只是"有路"，资源还得**有公网 IP**，IGW 才能真正帮它收发。

---

## 3. 公有子网"双向"的两个前提（必要≠充分）

路由表通 IGW 只是网络层"有路径"，真正双向还要：
1. **资源有公网 IP**：EC2（有公网IP）能双向 ✅；**Lambda（只有私有IP）就算在公有子网也上不了网** ❌。
2. **安全组放行入站**：安全组默认拒绝入站，外部要能访问必须显式放行端口。

> 公有子网 = 让双向**成为可能**；能否真双向 = 路由表(路径) + 公网IP(地址基础) + 安全组(最后放行)。

---

## 4. NAT：让私有子网"只出不进"地上网

- 私有子网把 `0.0.0.0/0` 指向 **NAT**（而非 IGW），就能让里面资源**主动出网**，但外部**无法主动进来**（单向）。
- **NAT 必须部署在公有子网里**，它靠自己的公网 IP + 那个子网的 IGW 路由上网：
  ```
  私有子网资源(私有IP) → NAT(公网IP, 在公有子网) → IGW → 公网
  ```
- NAT 两种：**NAT Gateway**（托管、高可用、约$32/月）｜ **NAT 实例**（自建 t2.micro EC2、免费套餐内、但要自己维护、是单点）。

## 5. Lambda 进 VPC 的网络（区分"进来"和"出去"）

| 方向 | 是什么 | 受 IP/VPC 影响吗 |
|------|--------|------------------|
| **进来**：被触发（API Gateway 等调用） | 走 **AWS Lambda 调用 API**，不通过 IP 连接 | ❌ 不受影响，VPC 里照样能被触发 |
| **出去**：函数代码主动外连（调 GitHub） | 走网络，需要公网路径 | ✅ 受影响：私有IP → 需要 NAT |

- "Lambda 进 VPC 只有私有 IP" 只影响**出去**，不影响**被触发**。
- 触发 Lambda 从不靠"连它的 IP"，而是调用 Lambda 服务的 Invoke API。

### 省 NAT 的"两 Lambda"模式
- **Lambda A（VPC外）**：调 GitHub（要公网）→ 再 `invoke` B。
- **Lambda B（VPC内, 无NAT）**：只读写私有 RDS（不上网，所以不需要 NAT）。GET 直接走 B。
- A 调 B 走 AWS Lambda API（VPC 外也能调）。结果：RDS 私有 + 无 NAT + 保留 MySQL + 全免费。

## 6. 安全模型：NACL vs 安全组

数据包进来：**先过 NACL（子网层）→ 再过安全组（资源层）**，两层都放行才通。

| 维度 | 安全组 (SG) | 网络 ACL (NACL) |
|------|-------------|-----------------|
| 作用层级 | 单个资源（网卡） | 整个子网 |
| 有状态? | ✅ 有状态：放行一个方向，回程自动通 | ❌ 无状态：来回都要手动放行 |
| 规则类型 | 只能"允许"（白名单） | "允许"+"拒绝"都行（能做黑名单） |
| 匹配方式 | 所有规则一起看，有允许即放行 | 按编号顺序，第一条命中即生效 |
| 默认 | 入站全拒、出站全放 | 入站出站全放行 |

### 最关键：有状态 vs 无状态 ⭐
- **安全组（有状态）**：放行入站 80 → 响应自动放行，**不用配出站**。
- **NACL（无状态）**：放行入站 80 → 响应往外发时它再查出站规则；响应走**临时端口(1024-65535)**，**忘了放行出站临时端口 = 入站通了却连不上**（经典坑）。

### 实践结论
- **安全组是主力、日常工具**；**NACL 是子网级粗粒度补充，大多数项目保持默认(全放行)**，只靠安全组把关。
- 动 NACL 的场景：整子网封禁某恶意 IP 段（SG 不能"拒绝"）、纵深防御。
- 比喻：SG = 每个人门口的私人保镖（贴身/有状态/只有宾客名单）；NACL = 小区大门检查站（管整个小区/来回都查/有黑白名单/按序检查）。

## 7. VPC 安全模型全景

```
路由表   → 决定流量「往哪走」（公私之分、能否出网）
公网 IP  → 决定资源能否真正用 IGW 收发
NACL    → 子网级防火墙（无状态、可拒绝、粗粒度，通常默认放行）
安全组   → 资源级防火墙（有状态、只允许、细粒度，主力把关）
```

### 一个请求完整走一遍（访问公有子网 EC2 的 80）
```
进：Internet → IGW → NACL入站(放行80?) → 安全组入站(放行80?) → EC2
回：EC2 → 安全组(有状态,自动放行) → NACL出站(放行临时端口?) → IGW → Internet
      ↑ NACL 出站要手动放行临时端口；安全组不用管
```
