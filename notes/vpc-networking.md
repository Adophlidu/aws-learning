# VPC 网络笔记

> 第 5 课 + 深入问答整理 ｜ 2026-06-28
> 从概念到真实电商架构，一路追问出来的立体理解。

---

## 一句话定位

**VPC = AWS 给你圈的一块隔离的私有网络（一块"地"）。你的 EC2、RDS 等资源都放在某个 VPC 的某个子网里。不同 VPC 默认互相隔离。**

---

## 核心层级

```
VPC（一块隔离的地，CIDR 如 10.0.0.0/16）
 └── 子网 Subnet（地里的区块 = 一段 IP 池 = 绑定一个 AZ）
      └── 资源（可放很多个：EC2 集群 / RDS / 缓存…）
```

| 概念 | 一句话 | 比喻 |
|------|--------|------|
| VPC | 你专属的隔离私有网络 | 整栋楼 / 一块地 |
| 子网 | VPC 内的网段，分公有/私有，绑定一个 AZ | 楼层 / 区块 |
| 公有子网 | 有通往 IGW 的路 → 能被公网访问 | 临街店面 |
| 私有子网 | 没有通往 IGW 的路 → 和公网隔绝 | 内部办公区 |
| 安全组 | 服务器的防火墙白名单（按"工牌"放行） | 门口保安 ⭐ |
| Internet Gateway (IGW) | 公网进出 VPC 的唯一大门 | 大楼正门 |
| NAT Gateway | 私有子网"只出不进"访问外网 | 员工通道（能出不能被外人进） |

---

## 1. 子网：公有 vs 私有的真正区别

- **同一个 VPC 内的所有子网（不分公有/私有）默认就能互相访问**（VPC 自带"本地路由 local route"）。
  - "内部" = 同一个 VPC 里的所有资源，不是只指私有子网。
- **公有/私有的本质区别不在子网本身，而在路由表有没有指向 IGW 的那条路：**
  - 有路 → 能和公网双向通信 → 公有
  - 没路 → 和公网隔绝、外网没有路径进来 → 私有
- 数据库放私有子网更安全：**不是它拒绝你，而是压根没有从公网到它的路。**

## 2. 子网容量：一个子网能装很多资源

- 子网 = 一段 IP 地址池。每个资源占一个 IP。
  - `/24` ≈ 251 个可用 IP（AWS 每个子网保留 5 个）
  - `/16` ≈ 6 万多
  - `/28` ≈ 11 个
- 所以**一个子网可以放一整个集群**，不是 1:1。
- 但子网**绑定单个 AZ**，所以生产中让集群**跨多个 AZ 的子网分布**做容灾。
  - "跨 AZ" 本质 = 在多个 AZ 各建一个同角色的子网。

## 3. 安全组：信任链（按"工牌"放行，不是按 IP）⭐

- 安全组 = 绑在资源上的防火墙白名单。默认全拒绝，需显式开放。
  - 入站 Inbound：谁能连进来（最常配）
  - 出站 Outbound：能连出去哪（默认全开）
- 常用端口：22(SSH) / 80(HTTP) / 443(HTTPS) / 3306(MySQL) / 5432(PG)
- **高级用法：安全组规则可以引用"另一个安全组"，而不是 IP。**
  - 形成信任链：公网 → ALB-SG → GraphQL-SG → 业务服务-SG → 数据库-SG，每环只信上一环。
  - 好处：不用关心对方 IP（IP 会变），按"挂着哪个安全组（工牌）"放行。
- ⚠️ 新手最经典的坑：EC2 起来了网页打不开 → 90% 是安全组没放行对应端口。

## 4. 寻址 vs 授权：两件不同的事，都不靠裸 IP

| 关注点 | 解决什么 | 由谁负责 |
|--------|----------|----------|
| **授权** 能不能连进来 | 谁有资格访问我 | 安全组（按工牌，不看 IP） |
| **寻址** 往哪个地址连 | 请求发到哪 | 稳定域名 / 负载均衡器 / 服务发现（不用裸 IP） |

- 服务 IP 会变（扩容/重启/换机器），所以没人连裸 IP。
- **寻址机制：**
  - 内部负载均衡器（Internal ALB）：有固定 DNS 域名，自动转发到当前健康实例。
  - 服务发现（AWS Cloud Map / ECS/EKS）：注册逻辑名字 `order-service.internal`，DNS 自动跟随实例更新。
- 比喻：公司搬家（IP 变）→ 靠电话/域名找到它（寻址）→ 到前台查工牌放行（授权）。两者都不需要知道它在哪栋楼。

## 5. 真实电商架构（把概念串起来）

组件：网站前端、GraphQL 聚合服务、业务微服务（用户/订单/商品）、数据库+缓存。

```
                       🌍 公网用户
                           │ https
                           ▼
                   Internet Gateway (IGW)
                           │
┌────────────────── VPC 10.0.0.0/16 ──────────────────┐
│  公有子网(跨AZ)：ALB（唯一对公网入口）+ 可选 Bastion 跳板机 │
│         │ 只允许 ALB 转发                              │
│  私有子网·应用层(跨AZ)：GraphQL + 用户/订单/商品服务(各成集群)│
│         │ 只允许应用层连                               │
│  私有子网·数据层(跨AZ)：RDS/Aurora + Redis             │
│  （私有子网访问外网 → NAT Gateway，只出不进）            │
└──────────────────────────────────────────────────────┘

🌐 静态前端通常不在 VPC：放 S3 + CloudFront(CDN)，前端再调 ALB 的 API
```

**设计思想：像剥洋葱，越往里越敏感越受保护。每层只对上一层开放。**

**请求流（查订单详情）：**
1. 浏览器 → IGW → ALB（公有子网，443 放行）
2. ALB → GraphQL（SG 只放行 ALB-SG）
3. GraphQL 并行调 订单/商品/用户服务（走 VPC 内部本地路由；各服务 SG 只放行 GraphQL-SG）
4. 各服务查 RDS/Redis（数据库 SG 只放行 业务服务-SG）
5. GraphQL 聚合结果 → ALB → 用户

**黑客视角（体会安全性）：**
- 直连数据库？私有子网无 IGW 路径 → 没路 ❌
- 绕过 GraphQL 打业务服务？私有子网 + SG 只信 GraphQL-SG ❌
- 只能打到 ALB（唯一暴露面）→ 在这一层集中防护（WAF/限流/HTTPS）

**几个精度点：**
- ALB 给应用层 HTTP 用；**数据库不挂 ALB**，用自己的集群端点（Aurora 的 Writer/Reader Endpoint）分发读写。
- 同 VPC 通信 = 快 + 私密（走 AWS 内网不经公网）+ 便宜（同 AZ 免流量费，跨 AZ 极少费用）。
- 这就是"相关服务放同一 VPC"成为标准做法的原因。

## 6. 创建资源时怎么选网络

- 创建 EC2/RDS 时**主动选「VPC + 子网」**（子网隐含了 AZ 和公有/私有）。
  - EC2 → 选一个具体子网。
  - RDS → 要一个跨 AZ 的"DB 子网组"。
- **一个资源只属于一个 VPC，创建时定死**，想换一般要重建。
- **不同 VPC 默认完全隔离**，要互通需 VPC Peering / Transit Gateway。
- 账号自带**默认 VPC**（每个 AZ 预置公有子网），所以新手"不选也能开机"。

---

## 整体心智图

```
VPC（一块隔离的地：内部互通、私密、低延迟）
 ├── 公有子网(多AZ) → 对外 ALB（HTTP 唯一入口）
 ├── 私有子网·应用(多AZ) → 服务集群（前面可挂内部LB / 服务发现）
 └── 私有子网·数据(多AZ) → 数据库集群（自带读写端点）

寻址靠稳定域名，授权靠安全组，二者都不依赖裸 IP
→ 机器随便增减/换 IP，系统照常运行（又一次"加一层抽象解决问题"）
```
